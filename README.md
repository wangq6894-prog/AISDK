# AISDK

大二下学期做的一个项目。想写个能跟 AI 聊天的网页，就搞了这个——底下接 DeepSeek 和 Ollama，上面搭了个 HTTP 服务。

前后一个多月，能跑。ChatGPT 和 Gemini 的框架写了但没调通，暂时没配环境就先搁那了。

## 干了什么

### 统一模型接入

不同模型的调用方式不一样——请求格式、返回格式、鉴权方式都不一样，如果每个模型单独写一套逻辑，加一个模型就要改一大片代码。

所以写了个基类 LLMProvider，规定了四个方法：初始化（传 API Key 这些配置进去）、检查是否可用、同步发消息、流式发消息。DeepSeek 和 Ollama 各写一个子类继承它，各自实现跟对应 API 通信的细节。之后想加新模型的话，继承基类实现这四个方法然后注册一下就行，别的地方一行业务代码都不用改。

框架其实还写了 ChatGPT 和 Gemini 的子类，但 sendMessage 和 sendMessageStream 没调通，暂时用不了。

### 流式输出

就是那种打字机效果，AI 一个字一个字往外蹦。底层用的是 SSE 协议。

大概流程：前端发消息过来，后端向模型发起一个 HTTP 长连接请求。模型那边生成一点内容就返回一点，后端这边收到一块数据就解析一块、推给前端一块。数据不一定每次刚好是一条完整的消息——TCP 是流式的，可能一个数据包里只有半条 JSON，也可能一个包里塞了两条。处理方式是把收到的数据先拼到一个 buffer 里，然后按双换行符切块（SSE 协议规定事件之间用空行隔开），切出来的完整块解析 JSON 取出新生成的文字片段推给前端，没切完整的留在 buffer 里等下批数据到了继续拼。直到收到结束标记为止。

### 接口

写了七个 REST 接口：建会话、查会话列表、查模型列表、删会话、看聊天记录、发消息（一次性返回）、发消息（流式返回）。

每个接口的处理流程都是一样的：解析请求里的 JSON、检查必填参数有没有、调 SDK 对应的方法、把结果拼成 JSON 返回。跨域头统一加了三个，不然前端调不了。

### 存数据

聊天记录用 SQLite 存，建了两张表，sessions 和 messages，messages 通过外键关联 sessions。服务重启了数据不会丢。

同时内存里用 unordered_map 也存了一份会话索引，读的时候先查内存，没有再查数据库，查到了就塞回内存。消息不是一次性全加载的——列举会话的时候不加载消息，只有实际查看某个会话的聊天记录时才去数据库取。相当于内存是缓存，数据库是兜底。

并发安全方面，session 管理那层和数据库操作那层各有一把互斥锁，增删改查都先加锁。session ID 用原子变量自增生成，确保不会重复。

### 工程化的东西

用 CMake 管理构建，SDK 编成静态库，服务端编成可执行文件。命令行参数用 gflags 处理，启动的时候可以指定端口号、温度参数、最大 token 数这些东西。API Key 从环境变量读，不写代码里也不写配置文件里。日志用 spdlog，分五个级别方便调试。

## 怎么跑起来

环境要求 Ubuntu 22.04，需要装 build-essential、cmake、OpenSSL、SQLite3、jsoncpp、spdlog、gflags、fmt。

先编译 SDK 那个目录，编出静态库之后装到系统目录。然后编译 ChatServer 目录，编出可执行文件。启动前先设环境变量 deepseek_apikey，然后运行 AIChatServer 指定端口号。用 curl 就能测试——先调建会话接口拿到 session id，再用这个 id 发消息。
